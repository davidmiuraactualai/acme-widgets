# ADR 0002: Dynamic Widget Catalog Backed by Google Sheets

- **Status:** Accepted
- **Date:** 2026-05-01
- **Deciders:** Bryan
- **Supersedes (in part):** [ADR 0001 — "Data" section](../decision-records/0001.ADR.TECH_STACK.md#data)

## Context

ADR 0001 chose a static CSV at `public/data/widgets.csv` as the widget
catalog. Producers and employees edit the catalog by opening a PR against
that file. That worked while the catalog was small and edits were rare,
but two friction points have surfaced:

1. **Edits require a deploy.** A widget price change or stock correction
   waits on a PR review, a CI build, and a Pages deploy. That is the right
   shape for code, not for data the business expects to update casually.
2. **Non-engineer producers cannot self-serve.** Editing CSV in GitHub is
   a poor experience for the producer/employee personas described in
   [`PERSONAS.md`](../current/PERSONAS.md), and we have no good way to
   teach trainees factory workflows around catalog updates without
   inventing one.

We want catalog edits to be live (visible to site visitors within minutes,
not on the next deploy) and to happen in a tool a non-engineer already
knows. We do not want to walk back any of ADR 0001's hard constraints:

- Fast trainee setup, no account provisioning required to run locally.
- Zero hosting cost; GitHub Pages stays as the host.
- No backend, no database, no paid services on the critical path.

## Decision

### Source of truth

The widget catalog moves to a single Google Sheet, "Acme Widgets — Catalog,"
with one row per widget and a header row whose column names match the
existing CSV (`type`, `description`, `number_in_stock`, `price`, `icon`,
`categories`). Producers and employees edit the sheet directly; engineers
no longer ship PRs to update prices or stock counts.

### Delivery to the browser

The sheet is **published to web as CSV** via Google Sheets'
`File → Share → Publish to web → CSV` feature. The published URL is a
stable, public, CORS-permissive endpoint of the form:

```
https://docs.google.com/spreadsheets/d/e/{publish-id}/pub?output=csv
```

The site fetches that URL at runtime from the browser, parses it with the
existing `papaparse` pipeline in `src/lib/widgets.ts`, and renders the
catalog. No API key, no OAuth, no backend, and no GCP project are
required. Google's edge serves the CSV; we treat it as a CDN we don't pay
for.

### Configuration

The published-CSV URL is provided to the build via a Vite env var
(`VITE_WIDGETS_SHEET_URL`) and read in `src/lib/widgets.ts`. The default
value points at the canonical Acme Widgets sheet so a fresh clone runs
out of the box without any setup. Trainees who want to point the example
at their own sheet override the env var locally.

### Fallback

`public/data/widgets.csv` stays in the repo as a **last-resort fallback**
only. If the Sheets fetch fails (network error, sheet unpublished,
Google outage), `fetchWidgets()` retries against the local CSV and logs
a warning to the browser console. The fallback is whatever is checked
into git at the time — Acme Widgets is a demonstration site, not a
production storefront, so the fallback existing matters more than it
being current. There is no scheduled refresh; if the snapshot drifts
from the live sheet, that is acceptable for the demo.

### Schema validation

Bad data shows up in two shapes, and they are handled asymmetrically:

- **Bad row** — a single widget is missing a required field (`type` or
  `icon`). The current behavior is preserved: skip the row and log a
  warning to the browser console. The rest of the catalog renders.
- **Bad header** — a column has been renamed in the sheet (e.g.
  `price` → `Price ($)`), which would silently corrupt that column for
  every widget. This fails loud: `parseWidgetsCsv` throws, and the
  catalog page renders a visible error state instead of garbage.

Validation runs at runtime in the browser. There is no scheduled CI
check against the live sheet — it would reintroduce the same kind of
scheduled ops work we declined for the fallback refresh.

### Local development

`bun dev` hits the live sheet by default, matching production behavior.
If the live fetch fails, the same fallback path applies and the warning
appears in the browser console — trainees working offline will see the
catalog from the local CSV with a clear signal in the console that
they're not on the live data.

### Caching and freshness

Google's published-CSV endpoint is edge-cached; in practice edits appear
within ~5 minutes. The browser-side module cache in `widgets.ts` is kept
as-is (one fetch per page load shared across views), so within a single
session a visitor sees a consistent snapshot.

## Consequences

**Positive**

- Producers edit prices and stock in a familiar spreadsheet UI; changes
  reach visitors in minutes without a PR or a deploy.
- The web app stays a static bundle on GitHub Pages. Hosting cost stays
  zero. No backend is introduced.
- The change is small and localized: a single fetch URL flips, the parse
  pipeline is unchanged, and the existing tests for `parseWidgetsCsv`
  remain valid.
- Trainees see a recognizable real-world pattern (Sheets-as-CMS) without
  the overhead of a real headless CMS.

**Negative / accepted trade-offs**

- The catalog is now publicly readable via the published-CSV URL. That
  is acceptable — the site already publishes the same data on `/widgets`
  — but it means the sheet must contain only catalog data, never internal
  notes.
- Anyone with edit access to the sheet can change live site content with
  no review. Mitigated by limiting sheet editors to the same people who
  could already merge PRs to `main`.
- A Google Sheets outage degrades the site. Mitigated by the local-CSV
  fallback.
- The "publish to web" URL format is undocumented as a stable contract.
  It has been stable for the better part of a decade, so we accept the
  risk; if Google changes the format, ADR 0003 revisits the choice.

## Alternatives considered

- **Google Sheets API with an API key.** Rejected: requires a GCP
  project, billing-enabled or not, and an API key in env. Conflicts with
  ADR 0001's "fast to set up, no account provisioning" constraint.
- **Build-time pull from the sheet via GitHub Actions, commit the CSV.**
  Rejected: edits still wait on a CI run, which defeats the "live" goal.
  Also rejected as a *fallback refresh* mechanism — Acme Widgets is a
  demonstration site, and a static fallback that drifts from the live
  sheet is acceptable for the demo.
- **Headless CMS (Sanity, Contentful, Strapi).** Rejected: each
  introduces an account, a free tier with a billing risk, and a
  modeling concept (schemas, references) that is overkill for a flat
  list of widgets and noisy for the training scope.
- **Airtable.** Rejected: similar shape to Sheets but with a more
  restrictive free tier and an editor experience trainees are less
  likely to recognize.
- **Cloudflare Worker (or similar) proxying the Sheets API.** Rejected:
  introduces a backend service on the critical path, which ADR 0001
  rules out.
- **Firebase / Firestore.** Rejected for the same reasons given in ADR
  0001 — account setup, billing risk, real database out of scope.

## Open questions

- **Test strategy.** Current tests parse a fixture CSV string. The
  fetch path needs a test against a stubbed Sheets response and a test
  for the fallback when the fetch fails. Out of scope for this ADR;
  capture in tasks under the implementation plan.

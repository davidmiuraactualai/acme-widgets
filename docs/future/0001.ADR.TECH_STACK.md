## Tech Stack

Acme Widgets is an example website for use in instructional exercises at the Software Factories Intensive training. It should be fast and easy to set up and use. And it should take a advantage of free services wherever possible.

The technical starting-point for the example website running on main is described below.

Use:
- React with Typescript
- Bun
- Tailwind
- React Router

## DOCS

Any mature high-performing software factory:
- Agents read and write all the code; humans read and write in plain English
- Agents do code reviews; humans review functionality and plain English descriptions in docs/


The basic directory structure for docs looks like this:
docs/current/
docs/future/
docs/roadmap → docs/future
docs/decision-records/
docs/human-review-logs/

Envisioned improvements get drafted and proposed here as ADRs:
docs/future/

Symlink for anyone looking for "the roadmap":
docs/roadmap → docs/future

After an ADR is approved and a branch is created to implement it, the ADR is moved here, to be included in a pull request to main when the work is completed. If architecture decisions evolve during implementation and before merge, Those updates should be made in the ADR here:
docs/decision-records/

After the merge, the ADR is a historical record of a decision made a point in time in a particular context, it doesn't change. New decisions will get new to ADRs. Meanwhile, technical documentation reflective of the current state of the code base lives here:
docs/current/

Any pull request that makes a meaningful change to technical architecture, user experience design, etc. should be summarized in changes/updates to corresponding documentation in docs/current.

The other update to docs that should be included where appropriate for human review is the review log. This is a list of checkboxes for suggested functional reviews to be performed by a human. Where are these reviews corresponded with an automated test, that test should be referenced for context. The idea with these reviews is to make it easy for a human to verify as efficiently as possible that intended changes have been made and deliver the intended results.
docs/human-review-logs/<bead-id>--<slug>.md

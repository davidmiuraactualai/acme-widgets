# ADR 0001: Tech Stack & Hosting for Acme Widgets

- **Status:** Accepted
- **Date:** 2026-04-29
- **Deciders:** Bryan

## Context

Acme Widgets is the example website used in the Software Factories Intensive
training. It exists to give trainees a small, realistic codebase to practice
factory workflows on. That purpose drives three constraints:

1. **Fast to set up.** A trainee should be able to clone, install, and run
   locally in minutes, on a clean laptop, without account provisioning.
2. **Free to operate.** No paid services on the critical path. The repo should
   stay green on a personal GitHub account at zero cost.
3. **Recognizable to working engineers.** The stack should reflect choices a
   real product team might make in 2026, so lessons transfer.

The site itself is small: a marketing home page, a static widget catalog, and
a placeholder orders page. There is no auth, no database, no payments — those
are intentionally out of scope so the example stays focused on factory
mechanics rather than product complexity.

## Decision

### Application stack

- **Language:** TypeScript
- **UI framework:** React
- **Runtime / package manager:** Bun
- **Bundler:** Vite (with Bun as the package manager and script runner)
- **Routing:** React Router (data router, v7)
- **Styling:** Tailwind CSS

### Data

> **Superseded in part by [ADR 0002](./0002.ADR.DYNAMIC_WIDGET_CATALOG.md):**
> the widget catalog moved from a static CSV to a Google Sheet published as
> CSV, fetched live by the browser. The bullet below describes the original
> decision and is preserved as historical context.

- **Widget catalog:** static CSV checked into the repo at
  `public/data/widgets.csv`, fetched at runtime by the catalog page and parsed
  client-side. Producers and employees update the catalog by editing the CSV
  and opening a PR.
- **No backend, no database.** Orders are handled out-of-band by emailing
  `sales@acmewidgets.com`.

### Hosting & CI

- **Hosting:** GitHub Pages (free for public repos), serving the static build
  output.
- **CI:** GitHub Actions (free tier for public repos) for build, test, and
  deploy-to-Pages on merge to the default branch.

### Testing

- **Unit / component tests:** Vitest (Jest-compatible API, runs natively under
  Bun/Vite without the Node-compat overhead Jest requires).
- **End-to-end tests:** Playwright, headless in CI.

> Note: the rough notes proposed Jest. Vitest is preferred because it shares
> Vite's transform pipeline and works with Bun without extra configuration.
> If a future ADR moves the project off Vite, revisit this choice.

## Consequences

**Positive**

- Zero hosting cost; trainees can fork and deploy on their own GitHub account.
- Bun + Vite gives sub-second cold starts on local dev, keeping the inner loop
  tight during exercises.
- Static CSV catalog means no database, no migrations, no seeding scripts —
  exercises can focus on factory workflow rather than infrastructure.
- TypeScript + React + Tailwind matches what most working engineers use today,
  so trainees recognize patterns immediately.

**Negative / accepted trade-offs**

- GitHub Pages serves static assets only. Any future feature requiring a
  server (real orders, search, auth) will require a new ADR and likely a
  hosting change.
- CSV-as-database does not scale past a few hundred rows or any concurrent
  writers. That is acceptable for the training scope.
- Bun is younger than Node; some npm packages have rough edges under it.
  Mitigated by keeping dependencies few and mainstream.

## Alternatives considered

- **Next.js on Vercel.** Rejected: requires a Vercel account, and SSR/ISR
  features add concepts the training doesn't need.
- **Plain HTML + a static site generator (Astro, 11ty).** Rejected: trainees
  expect React; SSGs add a meta-framework concept that distracts from factory
  exercises.
- **Firebase Hosting + Firestore.** Rejected: introduces account setup,
  billing risk, and a real database — all out of scope for the example.
- **Jest for unit tests.** Rejected in favor of Vitest, see Testing above.

### Default branch

`main`. Acme Widgets has no need for parallel major-version branches and
follows the GitHub default.

## Open questions

- ~~Should the CSV be parsed at build time (typed import) or fetched at
  runtime (smaller bundle, easier to hot-swap)?~~ **Resolved during
  implementation:** runtime fetch via `papaparse`, with the parsed list
  cached in module scope so the home featured trio and the catalog page
  share one network request. See `src/lib/widgets.ts`.
